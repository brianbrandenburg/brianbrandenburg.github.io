Data Admin

Visualizing data is all about... the data. So, the Data Admin aspect of the product was very important. In Zoomdata, Data Admin was a series of steps that guided users through connecting to a source, selecting a view, enriching that view, and ultimately configuring default visualizations. The focus was on data exploration, which translated the end user a chart as quickly as possible.

  • Company: Zoomdata
  • Product: Zoomdata
  • Completed: 10 Sept 2019

The Challenge

The Data Admin's user interface was a mixture of React and Backbone which had lay dormant for several years. The Dev team had been wanting to rewrite these screens in React for a while, but the complexity of the code base combined with a rapid release model created challenges:

  1. Do you take this one panel at a time, slowly introducing inconsistency over multiple sprints while you work on other features.
  2. Do you embrace a lengthy dev cycle, and rewrite all the panels at once so that you don't introduce inconsistencies.

The Solution

Product and stakeholders did not like the sound of a lengthy refactor. They wanted small, bite sized tickets that were easier to point. However, they didn't want to introduce inconsistencies within the existing Data Admin while we iteratively worked through these screens. I suggested a third option. I'd personally go through and restyle the existing panels to look like the targeted React components being used in other parts of the product. This would provide a visual scaffolding, allowing dev teams to rewrite these panels in an incremental fashion without introducing one off inconsistencies within the overall flow. The examples below show what I did. The implementation took me one sprint (2 weeks). It required ~6,000 lines of CSS code due to the way the original CSS was written.

Step 1: General

The general step involved transitioning from floats to a flexbox layout, styling inputs to look like blueprint components, and introducing some consistency around when we use primary buttons. If every is important, then nothing is.

Step 4: Enrich

There were lots of peripheral dialogs and controls off of this view, but it's essentially three separate tables showing Raw, Derived, and Calculated fields. I wanted to condense this down to a single table to make finding fields by name a little simpler. The hypothesis; A user shouldn't have to know how a field was generated in order to find it. This wound up substantially increasing scope, so I kept the original three table layout in order to stay on schedule.
