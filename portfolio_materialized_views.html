Materialized Views

The purpose of a materialized view is to increase performance and reduce load on the system. Instead of re-running a common or expensive query, the product would detect the presence of a materialized view and just point to an existing result set instead.

  • Company: Zoomdata
  • Product: Zoomdata
  • Completed: 17 Aug 2018

Challenge

When I started on the Materialized View feature, the backend design had already been completed. It spelled out exactly what the front end had to do, but did not consider what the end user experience would be.

Original Design

I always start from baseline assumptions, regardless of whether I agree with the overall approach or not. In this case, I mocked up what the front end would look like based on the back end requirements I'd been given:

  1. Paste a query that represents a materialized view, an existing view of the data that exists somewhere.
  2. Parse the query, and Generate a list of mappings between the current source and the targeted materialized view.
  3. Do a lot of complex validation to ensure that field and table names match between the two sources.

The mocks above show what I considered to be a rather disjointed UX, so I decided to try out a few alternatives.

Proposed Design

The concepts below show a revised approach centered around a visualization. Rather then copy/paste an abstract query as a starting point, why not just provide an option off a visualization?

Result

The visualization already knew what the query parameters were needed. This removed the need for user from to copy/paste a SQL query. The visualization simply provided the things that had to be mapped in order for the materialized view to work. This approach streamlined the entire flow and reduced the amount of client development required.
