Materialized Views

The purpose of a materialized view is to increase performance and reduce load on the system. Instead of re-running a common or expensive query, the product would detect the presence of a materialized view and just point to an existing result set instead.

When I started on the Materialized View feature, the backend design had already been completed. It spelled out exactly what the front end had to do in order to get the backend to work. The problem was that the back end design resulted in a complex front end.

  • Company: Zoomdata
  • Product: Zoomdata
  • Completed: 17 Aug 2018

Original Design

The original design required an Admin user to:

  1. Paste a query that represents a materialized view, an existing view of the data.
  2. Generate a list of mappings between the current source and the materialized view.
  3. Do a lot of complex validation to ensure that field and table names match between the two sources.

The mocks above show what I considered to be a rather disjointed UX, resulting in a complex mapping UI.

Proposed Design

Since I had concerns with the original design, I also provided an alternative approach. Rather then copy/paste an abstract query, why not just provide an option off a visualization which already contains a query definition?

In this approach, there's no need to copy/paste SQL. The visualization provides query parameters. This also means that one side of the mapping exercise is a constant, values already known in our system. If we required Materialized Views to be registered as Data Sources in our system as well, then mapping field and table names becomes similar to other concepts that already exist in the product.
